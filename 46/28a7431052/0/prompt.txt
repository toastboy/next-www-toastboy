<result>I've fetched the official documentation. Here are precise answers to your five questions:

## 1. **Compound command parsing: decomposition**

**Yes, commands are decomposed.** The permissions doc states explicitly:

&gt; "Claude Code is aware of shell operators, so a rule like `Bash(safe-cmd *)` won't give it permission to run the command `safe-cmd &amp;&amp; other-cmd`. The recognized command separators are `&amp;&amp;`, `||`, `;`, `|`, `|&amp;`, `&amp;`, and newlines. **A rule must match each subcommand independently.**"

When you approve a compound command with "Yes, don't ask again", Claude Code saves separate rules for each subcommand, not one combined rule for the full string.

## 2. **Wildcard semantics**

Wildcards are **position-agnostic** (`*` can appear at start, middle, or end) and match "any sequence of characters including spaces." The **space-before-`*` matters**: 
- `Bash(ls *)` requires a word boundary, matching `ls -la` but **not** `lsof`
- `Bash(ls*)` without space matches both

The `*` character itself has no special escaping for shell metacharacters—it's a literal glob wildcard, not a regex. No documented grammar spec beyond "glob-like" is provided.

## 3. **Specific pipeline patterns**

**Not feasible with permission rules alone.** Since `find ... | xargs grep` is decomposed on the `|` operator, you'd need:
- `Bash(find *)` to allow find
- `Bash(xargs grep *)` to allow that xargs subcommand

However, **there's a special wrapper rule**: bare `xargs` (with no flags) is stripped during matching, so `Bash(grep *)` *already* matches `xargs grep pattern`. But the pipe still creates a compound command, so the full pipeline would decompose and require both rules.

## 4. **Security caveat**

**Yes, documented.** The permissions doc includes a **Warning** section titled "Bash permission patterns that try to constrain command arguments are fragile" and explicitly recommends:

&gt; "For more reliable URL filtering, consider… **Use PreToolUse hooks: implement a hook that validates URLs in Bash commands and blocks disallowed domains**"

This confirms permission patterns are a friction-reduction mechanism, not a sandbox boundary.

## 5. **PreToolUse hook alternative**

**Yes, viable.** Hook input/output contract (from hooks doc):

**Input (stdin JSON):**
```json
{
  "tool_name": "Bash",
  "tool_input": { "command": "find ... | xargs grep ..." },
  "cwd": "...",
  "permission_mode": "..."
}
```

**Output (stdout JSON, exit 0):**
```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "allow" | "deny" | "ask" | "defer"
  }
}
```

Exit codes: `0` = JSON decision parsed; `2` = blocking error (tool blocked). You can run a regex to accept only `find ... | xargs grep -l ...` (reject extra pipes, `&amp;&amp;`, `;`, backticks) and return `"allow"`, otherwise `"defer"`.

**Critical caveat:** "Hook decisions don't bypass permission rules. A matching deny rule blocks the call and a matching ask rule still prompts even when the hook returned `'allow'`." So the hook enhances but cannot override system deny/ask rules.

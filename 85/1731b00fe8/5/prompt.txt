Module augmentation is usually better placed in a dedicated *.d.ts file (e.g., src/mantine.d.ts) rather than a runtime module like theme.ts. This avoids relying on theme.ts being imported for types to take effect, and prevents the augmentation file from being treated as runtime code in bundling/test setups.

---

This assumes theme.other is always defined; if it is ever undefined at runtime, theme.other.appShellMaxWidth will throw a TypeError before your typeof checks run. Consider guarding theme.other first (e.g., const other = theme.other ?? {}) and validating against other.

---

If the test fails before the cleanup lines run, console.error and the mocked useMantineTheme return value can leak into other tests. Prefer restoring in a try/finally block or afterEach, and consider using mockReset()/mockRestore() consistently for the useMantineTheme mock.
